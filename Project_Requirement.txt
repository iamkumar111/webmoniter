## 1. Introduction

### 1.1 Purpose

This Software Requirements Specification (SRS) document describes the complete requirements for the Website Uptime Monitoring System. The system monitors website availability, SSL certificates, response times, and error logs from a single server with email alerts and role-based access control.

### 1.2 Scope

The system provides:
- Website uptime monitoring (HTTP/HTTPS ping)
- SSL certificate expiration tracking
- Response time measurement and tracking
- Error detection and logging
- Email alert notifications
- Role-based admin panel with 4 user roles
- Real-time dashboard with statistics
- Historical reporting and analytics

**Out of Scope:**
- Distributed monitoring from multiple locations
- SMS or mobile push notifications
- Integration with third-party monitoring services
- Public status pages

### 1.3 Definitions and Acronyms

- **Monitor:** A configured website or endpoint to check
- **Check:** A single monitoring attempt
- **Incident:** A period when a monitor is down
- **Uptime:** Percentage of time a monitor is operational
- **Response Time:** Time taken for a request to complete
- **SSL/TLS:** Secure Socket Layer / Transport Layer Security
- **RBAC:** Role-Based Access Control
- **SMTP:** Simple Mail Transfer Protocol

### 3.2 Monitor Management

**REQ-MON-01:** System shall allow creating monitors with:
- Monitor name (required)
- Target URL or IP address (required)
- Monitor type: HTTP, HTTPS, or Ping (required)
- Check interval: 1, 2, 5, 10, 15, 30 minutes, or 1 hour (required)

**REQ-MON-02:** For HTTP/HTTPS monitors, system shall support:
- HTTP methods: GET, POST, PUT, DELETE, HEAD
- Custom HTTP headers
- Request body for POST/PUT
- Request timeout (5-60 seconds)
- Expected HTTP status code (default: 200)
- Expected response content (keyword or regex)
- Follow redirects (yes/no)
- Verify SSL certificate (yes/no)
- Response time threshold for degraded status

**REQ-MON-03:** System shall allow configuring failure and recovery thresholds:
- Failure threshold: consecutive failures before marking as down (default: 1)
- Recovery threshold: consecutive successes before marking as up (default: 1)

**REQ-MON-04:** System shall provide monitor list with:
- Search by name or URL
- Filter by status (up, down, degraded, paused)
- Sort by name, status, response time, uptime

**REQ-MON-05:** System shall allow users to:
- Edit monitor configuration
- Delete monitor (with confirmation)
- Pause/resume monitoring
- Test monitor immediately (run check now)

**REQ-MON-06:** System shall display monitor status:
- **Up:** Responding correctly (green)
- **Down:** Not responding or error (red)
- **Degraded:** Responding but slow (yellow)
- **Paused:** Monitoring disabled (gray)

### 3.3 Automated Monitoring Engine

**REQ-ENG-01:** System shall run background process to check all active monitors automatically

**REQ-ENG-02:** System shall execute checks according to each monitor's configured interval

**REQ-ENG-03:** For each HTTP/HTTPS check, system shall:
- Send HTTP request using configured method and headers
- Measure response time in milliseconds
- Record HTTP status code
- Capture response headers
- Capture response body (first 1000 characters)
- Validate against expected status code
- Check for expected content if specified
- Check SSL certificate if HTTPS

**REQ-ENG-04:** For each ping check, system shall:
- Attempt TCP connection to target
- Measure connection time
- Detect connection failures

**REQ-ENG-05:** System shall store check results including:
- Timestamp
- Status (up/down/degraded)
- Response time
- HTTP status code (if applicable)
- Error message (if failed)
- Response headers and body

**REQ-ENG-06:** System shall update monitor status based on thresholds:
- Increment failure counter on each failed check
- Increment success counter on each successful check
- Reset failure counter when check succeeds
- Reset success counter when check fails
- Mark as DOWN when failures reach failure threshold
- Mark as UP when successes reach recovery threshold (after being down)
- Mark as DEGRADED when response time exceeds configured threshold

**REQ-ENG-07:** System shall handle errors gracefully:
- Network timeouts
- DNS resolution failures
- SSL handshake failures
- Connection refused errors
- Invalid responses

**REQ-ENG-08:** System shall limit concurrent checks to 10 to prevent server overload

**REQ-ENG-09:** System shall queue additional checks when concurrent limit reached

**REQ-ENG-10:** System shall retry failed checks once before recording as failure

---

### 3.4 SSL Certificate Monitoring

**REQ-SSL-01:** System shall automatically check SSL certificates for all HTTPS monitors

**REQ-SSL-02:** System shall extract certificate information:
- Valid from date
- Valid until date (expiration)
- Days remaining until expiration
- Certificate issuer
- Certificate subject (domain)
- Serial number

**REQ-SSL-03:** System shall calculate days remaining until certificate expires

**REQ-SSL-04:** System shall mark certificate status:
- **Valid:** More than configured warning days remaining
- **Expiring Soon:** Within warning threshold (default: 14 days)
- **Expired:** Past expiration date

**REQ-SSL-05:** System shall allow configuring SSL warning threshold (7, 14, or 30 days)

**REQ-SSL-06:** System shall display certificate information on monitor details page

**REQ-SSL-07:** System shall trigger alerts for certificate expiration warnings

### 3.5 Email Alert System

**REQ-ALERT-01:** System shall allow configuring email recipients for each monitor (comma-separated list)

**REQ-ALERT-02:** System shall send "Monitor Down" alert when:
- Monitor status changes to DOWN
- Failure threshold is reached

**REQ-ALERT-03:** "Monitor Down" alert shall include:
- Monitor name and URL
- Down since timestamp
- Number of consecutive failures
- Last error message
- Last response time
- Link to view monitor

**REQ-ALERT-04:** System shall send "Monitor Recovered" alert when:
- Monitor status changes from DOWN to UP
- Recovery threshold is reached

**REQ-ALERT-05:** "Monitor Recovered" alert shall include:
- Monitor name and URL
- Total downtime duration
- Recovered at timestamp
- Current response time
- Link to dashboard

**REQ-ALERT-06:** System shall send "SSL Certificate Expiring" alert when:
- Certificate days remaining reaches warning threshold
- Once daily until renewed

**REQ-ALERT-07:** "SSL Certificate Expiring" alert shall include:
- Monitor name and URL
- Certificate expiration date
- Days remaining
- Certificate issuer
- Link to certificate details

**REQ-ALERT-08:** System shall send "Slow Response" alert when:
- Response time exceeds configured threshold
- Alert enabled for the monitor

**REQ-ALERT-09:** "Slow Response" alert shall include:
- Monitor name and URL
- Current response time
- Configured threshold
- 24-hour average response time
- Link to performance data

**REQ-ALERT-10:** System shall use HTML email templates with:
- Responsive design
- Color-coded status indicators
- Action buttons with links
- Plain text fallback

**REQ-ALERT-11:** System shall implement alert cooldown (minimum 5 minutes between same alert type for same monitor)

**REQ-ALERT-12:** System shall track email delivery status (pending, sent, failed)

**REQ-ALERT-13:** System shall retry failed emails up to 3 times with exponential backoff

**REQ-ALERT-14:** System shall allow users to enable/disable alert types per monitor:
- Alert on down
- Alert on recovery
- Alert on SSL expiration
- Alert on slow response

**REQ-ALERT-15:** System shall provide "Send Test Alert" function to verify email configuration

**PERF-02:** System shall complete single HTTP check within 30 seconds (including timeout)

**PERF-03:** Dashboard shall load within 2 seconds on standard broadband connection

**PERF-04:** System shall handle up to 10 concurrent checks without performance degradation

## 7. Glossary

**Monitor:** A configured endpoint (website/URL) that the system checks periodically

**Check:** A single monitoring attempt to verify if a monitor is up or down

**Incident:** A period when a monitor is in down status

**Uptime:** Percentage of time a monitor is in up status

**Downtime:** Total duration when a monitor is in down status

**Response Time:** Time in milliseconds for a request to complete

**Degraded:** Status when a monitor is up but response time exceeds threshold

**Failure Threshold:** Number of consecutive failures before marking monitor as down

**Recovery Threshold:** Number of consecutive successes before marking monitor as up

**Alert Cooldown:** Minimum time between sending similar alerts

**MTTR:** Mean Time To Recovery - average time to resolve incidents

**MTBF:** Mean Time Between Failures - average time between incidents

**SSL Certificate:** Digital certificate that authenticates website identity

**Certificate Expiration:** Date when SSL certificate becomes invalid

**SMTP:** Protocol used for sending emails

**JWT:** JSON Web Token used for user authentication

**Role-Based Access Control (RBAC):** Access control based on user roles
